昇腾 Da Vinci 架构的 Memory Hierarchy
在昇腾芯片中，最核心的计算引擎叫做 AI Core。一个 AI Core 内部的存储层级如下：

1. 最内层：Cube 单元内部寄存器 (Registers/Latches)
对应我们聊的：MAC 阵列内部的 3 个寄存器。
官方说明：昇腾的矩阵乘法单元叫 Cube（通常是一个 16x16x16 的 3D MAC 阵列）。在 Cube 内部，确实有极少量的硬件 Latch，用于在每个时钟周期锁住权重、输入和部分和，维持脉动阵列的运转。这部分对程序员是完全透明的。
2. 贴身保镖：L0 Buffer (L0A, L0B, L0C)
对应我们聊的：L1 SRAM / Local Buffer（专门喂给 MAC 阵列的缓存）。
官方说明：这是昇腾架构的一大特色！为了喂饱饥渴的 Cube 阵列，华为设计了三个专门的 SRAM 块，紧紧贴着 Cube：
L0A Buffer：专门存左矩阵（比如输入特征图 X）。
L0B Buffer：专门存右矩阵（比如权重 W）。
L0C Buffer：专门存矩阵乘法的结果和部分和（Partial Sum）。
特点：带宽极其恐怖，刚好能满足 Cube 每个时钟周期的吞吐需求。
3. 核心大管家：Unified Buffer (UB)
对应我们聊的：AI Core 内部的共享 SRAM。
官方说明：这是一个容量更大（比如几百 KB）的 SRAM。它不仅给 Cube 用，还给 AI Core 内部的 Vector 单元（算向量）和 Scalar 单元（算标量）用。
作用：它是数据进出 AI Core 的集散中心。
4. 芯片级共享：L2 Cache
对应我们聊的：多个 Core 共享的 L2 SRAM。
官方说明：在 AI Core 之外，整个昇腾芯片（包含几十个 AI Core）共享一个巨大的 L2 Cache（几 MB 到几十 MB）。用于缓解多个 Core 同时访问 HBM 的拥堵。
5. 片外主存：HBM / DDR (Global Memory)
就是我们最熟悉的显存/内存。

这是一个非常敏锐且深入的思考！你的直觉很棒，试图把抽象的数据流（IS/OS/WS）和底层的硬件结构联系起来。

不过，这里有一个**理解上的小偏差**：

L0A、L0B、L0C **并不是**用来让你“三选一”从而实现 IS（输入固定）、WS（权重固定）或 OS（输出固定）的。

实际上，在一个矩阵乘法 $C = A \times B$ 中，**L0A、L0B、L0C 是同时在工作的**，它们分别对应矩阵乘法的三个操作数：A（输入左矩阵）、B（权重右矩阵）、C（输出/部分和）。

硬件上的数据流（IS/OS/WS）是由 **Cube 内部寄存器的连线和控制方式**决定的（这通常在芯片硬件设计时就固定了，比如 TPU 是典型的 WS，昇腾也有其固定的脉动阵列数据流设计），而不是通过“选择使用哪一个 L0 Buffer”来决定的。不管用什么数据流，你都必须同时提供输入、权重并找个地方存输出。

那么，回到你最核心的疑问：**“既然 Cube 是固定的，为啥要分成三个不同的 L0 Buffer？复用一个大 Buffer（比如 UB）不行吗？单纯是为了传输时延吗？”**

不仅仅是为了时延，更核心的原因是**“极其恐怖的并发带宽需求”**和**“数据格式及精度的差异”**。我们可以从以下三个维度来拆解硬件设计的初衷：

### 1. 物理极限：SRAM 的端口冲突与超宽并发带宽（最核心原因）

想象一下，Cube 是一个 16x16x16 的吞吐巨兽，它在一个时钟周期内要完成 4096 次 MAC（乘加）运算。为了在**每一个时钟周期**喂饱它，硬件需要同时做四件事：
1. 读出一批 Input A
2. 读出一批 Weight B
3. 读出一批上一次的 Partial Sum C（如果是累加操作）
4. （计算完毕后）写入一批新的 Partial Sum C

这就要求在一个极短的时钟周期内，存储器必须支持 **3次读取 + 1次写入**。

如果你把它们都塞进**同一个**物理 SRAM 中（即复用一个 Buffer）：
标准的 SRAM 通常只有 1 个读写端口（1R1W）或者双端口。要想实现单周期 3R1W，就需要设计非常复杂的“多端口 SRAM”（Multi-ported SRAM）。多端口 SRAM 的电路面积极大、功耗极高，而且布线极其困难，根本不适合做大规模片上缓存。

**解决办法就是“物理隔离”**：华为把 SRAM 切分成 L0A、L0B、L0C 三个物理上完全独立的小块。这样，Cube 就可以在同一个时钟周期，毫无冲突地同时从 L0A 拿数据，从 L0B 拿权重，在 L0C 存取部分和。它们各自都有专属的超宽数据总线直连 Cube，完美避开了端口抢占。

### 2. 精度与位宽的差异（量体裁衣）

在深度学习的混合精度计算中，输入、权重和输出的数据类型往往是不一样的：
- **Input (L0A) / Weight (L0B)**：通常是低精度的，比如 FP16 或 INT8。
- **Partial Sum / Output (L0C)**：为了防止成千上万次乘加累积时发生溢出或精度截断，C 矩阵必须使用高精度，比如 FP32 或 INT32。

如果混用同一个 Buffer，由于每次读取的数据位宽（16-bit vs 32-bit）不一致，内存分配、寻址逻辑会变得极其复杂，且容易产生碎片。专门设计一个 L0C，可以直接在硬件层面把 L0C 的字长（Word Length）做成 32 位宽的，而 L0A 和 L0B 做成 16 位宽的，不仅控制逻辑简单，还节省了芯片面积和功耗。

### 3. 数据排布（Format）与流水线（Pipeline）的要求

- **特殊的数据排布**：为了让矩阵乘法算得快，数据在从 UB 进入 L0 前，通常会被转换为特殊的硬件格式（昇腾称为 Fractal 格式，即 zZ 格式）。因为 A 矩阵是“按行”参与计算，B 矩阵是“按列”参与计算，所以 A 和 B 在底层内存中的最佳切块和排布方式是完全不同的。分成独立的 Buffer，可以让硬件的地址生成器（Address Generator）各司其职，高效管理。
- **Ping-Pong 双缓冲流水线**：为了绝对不让 Cube 停下来等数据，DMA 需要一边让 Cube 计算，一边从 UB 往 L0 搬运**下一批**数据（即 Ping-Pong Buffer 技术）。如果 A、B、C 都在一个大 Buffer 里，同时进行三个方向的 Ping-Pong 异步搬移，还要兼顾 Cube 每个周期的读取，读写控制逻辑会爆炸。独立分开后，A、B、C 可以各自简单独立地维护自己的 Ping-Pong 流水线。

---

### 总结一下：

你的直觉方向是对的，把它们放在离 Cube 最近的地方确实是为了解决“时延和搬运成本”，但这只说对了一半。

- **Cube 内部的寄存器**：决定了硬件走的是 IS、OS 还是 WS 策略（决定了运算时“谁留在原地不动”）。
- **分开的 L0A / L0B / L0C**：不是为了适配不同的策略，而是为了 **提供单周期无冲突的恐怖读写带宽**、**适应不同精度的物理位宽**，以及 **极大地简化流水线控制逻辑**。

可以把 Cube 想象成一个巨大的水泥搅拌机；L0A、L0B、L0C 就是三条独立的高速传送带，分别运送沙子、水泥和水。如果你想让搅拌机以最大功率运转，你必须用三条传送带同时往里倒料；如果只用一条传送带（一个复用的 Buffer）轮流运沙子和水泥，搅拌机绝大部分时间都在空转等待。

为了让你直观感受到硬件是怎么“控制”数据流的，我们拿最经典的 **权重固定（Weight Stationary, WS）** 数据流来举个实战例子。

为了方便大脑想象，我们把 16x16 的 Cube 缩小成一个 **3x3 的 MAC 阵列**。
我们的任务是计算矩阵乘法：$C = A \times B$。
此时，A（3x3，输入）已经在 L0A 里，B（3x3，权重）已经在 L0B 里，C（初始为0）在 L0C 里准备接收。

---

### 第一步：硬件的物理连线（天生决定了数据流向）

硬件和软件最大的区别在于，数据流不是靠 `for` 循环实现的，而是靠**物理电线**连出来的。

在 3x3 的 MAC 阵列中，每个 MAC 单元内部有一个乘法器、一个加法器，以及三个极小的寄存器：`Reg_A`、`Reg_B`、`Reg_C`。
由于我们设计的是**权重固定（WS）**架构，硬件工程师在画芯片图纸时，连线是这样设计的：
1. **Reg_B（权重）**：只有接收端，没有输出端。数据从 L0B 灌进来后，就死死锁在里面，不传给任何人。
2. **Reg_A（输入）**：横向连线。左边 MAC 的 `Reg_A` 出口，连着右边 MAC 的 `Reg_A` 入口。像一条横向水平的传送带。
3. **Reg_C（部分和）**：纵向连线。上边 MAC 的 `Reg_C` 出口，连着下边 MAC 的 `Reg_C` 入口。像一条纵向垂直的瀑布。

这种物理连线，就是“硬件控制”的基础！

---

### 第二步：实战推演（Cycle by Cycle 的心跳脉动）

当软件发出一条极其简单的机器指令：`Cube_MAC(L0A_addr, L0B_addr, L0C_addr, 3x3)`，硬件状态机（FSM）被激活，开始按照时钟周期（Clock Cycle，简称 T）“打拍子”执行。

#### **准备阶段：加载权重（Load）**
*   **T = 0 到 T = 2**：从 L0B 中把矩阵 B 读出来，塞进 9 个 MAC 单元的 `Reg_B` 里。此时，每个 MAC 单元里都住进了一个固定不变的权重值。权重就位，阵列准备完毕。

#### **执行阶段：脉动计算（Compute）**
为了让矩阵乘法能够像接力一样进行，L0A 的数据不能一窝蜂全涌进去，必须**“斜着走”（Skewed Input）**。硬件地址生成器会自动把 L0A 的数据排成阶梯状喂进去。

*   **T = 3（心跳第一拍）：**
    *   A 矩阵的**第一行第一个元素 $A_{11}$**，进入阵列**左上角**的第一个 MAC。
    *   **左上角 MAC 动作**：$A_{11}$ 乘以它内部固定的 $B_{11}$，加上从上方传来的 0，得到结果 $C_{11}$（部分和）。
    *   此时其他 8 个 MAC 还在空转等数据。

*   **T = 4（心跳第二拍）：**
    *   **横向传送**：左上角的 $A_{11}$ 通过物理导线，传给了**它右边**的 MAC。同时，$A_{12}$ 从 L0A 新进入了左上角的 MAC。
    *   **纵向传送**：左上角算出的部分和 $C_{11}$，通过纵向导线，流到了**它下边**的 MAC。
    *   现在，随着波浪推进，有 3 个 MAC 正在同时工作，它们并行做着各自的乘加运算。

*   **T = 5（心跳第三拍）：**
    *   波浪继续向右、向下蔓延。
    *   第一行的输入 A 继续向右传，第二、第三行的输入 A 跟着阶梯涌入。
    *   纵向的 C 像滚雪球一样，一路乘、一路加。
    *   到了这一拍，整个 3x3 阵列**全部激活**，9 个乘加单元在同一个时钟周期内全部满载轰鸣！这就是 NPU 算力恐怖的时刻。

*   **T = 6 及以后（输出结果）：**
    *   随着波浪到达右下角，计算完成的完整 $C$ 元素，开始一个个从阵列的最底端掉出来。
    *   掉出来的 $C$，被硬件自动按顺序写进 **L0C Buffer** 里。

整个过程没有任何的软件跳转指令，全部靠每个时钟周期（Clock Edge）到来时，触发器（Flip-flop）将信号推向下一个逻辑门。就像是一群人排成方阵在玩传球游戏，每听到一声哨响，就把手里的球乘上自己的数字，加上上一个人的数字，然后传给下一个人。

### 对比一下：如果是“输出固定（OS）”，硬件是怎么控制的？

理解了上面，你就很容易反推出 OS（Output Stationary）是怎么回事了。

在 OS 架构（很多早期的加速器或特定算子使用）里，硬件图纸的**物理连线被改了**：
1. **Reg_C（部分和）**：没有输出连线！运算的中间结果（Partial Sum）死死锁在 MAC 内部，不断自我累加。
2. **Reg_A（输入）**：横向连线，像传送带。
3. **Reg_B（权重）**：纵向连线，像垂直的瀑布。

**执行过程**变成了：
L0A 在横着发牌，L0B 在竖着发牌。当 A 和 B 在某个 MAC 节点相遇（交汇）时，乘一下，累加到这个 MAC 内部的寄存器里。
等所有回合都算完了（通常需要走完内循环的 K 维），此时所有 MAC 肚子里憋着的就是最终完整的 C 矩阵。最后硬件再花几个时钟周期，把这 9 个算好的 C 结果“吐”出来，一起写到 L0C 里。

### 总结硬件的“控制”艺术

所以，你问硬件是怎么控制数据流的？

1. **底层物理空间（骨架）**：是由不可变的**金属连线方向**决定的。不同的连线造就了不同的流（IS / WS / OS）。
2. **外围调度（血管）**：是由 **L0A / L0B / L0C 地址控制器**决定的。硬件会自动计算偏移量，把原本四四方方的矩阵数据，故意“错开拍子（Skew）”，有节奏地喂入阵列的边界端口。
3. **驱动力（心脏）**：是**全局时钟信号（Clock）**。每一个时钟上升沿，数据就被强制往前推一格。没有任何软件判断语句（没有 `if`，没有 `while`），极致的粗暴换来了极致的吞吐率。

在这个极微观的世界里，一切都是空间和时间的舞蹈。时间上的循环（Loop），被硬件强行展开成了空间上的方阵。这就是 AI 芯片设计的精髓！

这个问题问得**太深刻了**！你直接击穿了软件编译器（Compiler）和硬件微架构（Microarchitecture）之间的那层“窗户纸”。

你提出的矛盾非常精准：**既然 Cube 内部的连线（WS/IS/OS）是焊死的，那为什么在 AI 编译器（比如 TVM、AKG、MLIR）的调度语言中，还会提供 `spatialize`（空间化）、`tensorize`（张量化）或者 `bind` 这样的软件原语（Primitive）去控制空间计算呢？**

答案的核心在于：**软件的 `spatialize` 并不是去“改变”硬件的连线，而是为了“适配”硬件的物理空间，以及在更上层的缓存（L0/UB）中“伪装（或者说模拟）”出各种数据流。**

我们可以从以下三个维度来彻底解开这个疑惑：

### 1. 将“时间”转换为“空间” (Time to Space Mapping)

在普通的 CPU 编程思维中，所有的 `for` 循环都是在**时间**上串行展开的。比如一个 16x16 的矩阵乘法，在 C 语言里是两个嵌套的 `for` 循环，执行 256 次。

但 Cube 是一个二维的物理阵列（比如 16x16），它在**空间**上同时拥有 256 个计算单元。

编译器看到嵌套的 `for` 循环时，它是懵的，它不知道怎么把这 256 次串行计算喂给 256 个并行的物理单元。这个时候，就需要软件工程师（或者 Auto-Tuner）使用 **`spatialize`（空间化调度原语）**。

`spatialize` 的作用是告诉编译器：
*“嘿！不要把这个 `for i in 0..15` 和 `for j in 0..15` 编译成串行的机器码。请把它们**空间化**，直接映射到 Cube 的二维 X 轴和 Y 轴阵列上！”*

**结论：** 软件的 `spatialize` 并没有改变 Cube 是 WS 还是 OS，它只是一个“**填模子**”的动作。硬件提供了一个固定形状（16x16）的模具，软件必须用 `spatialize` 原语把数据切成对应的形状“嵌”进去。这在很多编译器里也被称为 `Tensorize`（张量化）。

### 2. 软件可以在“上层存储”控制数据流！（宏观数据流）

这是最精彩的部分。
刚才我们说，Cube **内部寄存器**的数据流（比如 WS，权重固定）是焊死的。但是！**L0 Buffer 和 Unified Buffer (UB) 是完全由软件控制的！**

虽然在最底层的 16x16 微观世界里，硬件只能做 WS；但软件可以通过控制外围的数据搬运（Tiling 和 Loop Ordering），在宏观上实现 IS（输入固定）或 OS（输出固定）。

**举个真实的例子：**
假设我们要算一个极大的矩阵乘法（4096 x 4096），而 Cube 一次只能算 16x16，这就需要软件写多层循环切块（Tiling）。

*   **软件如何实现“宏观的 OS（输出固定）”策略？**
    软件控制 DMA，把一个 128x128 的 $C$ 矩阵块分配在 **L0C Buffer** 中（不写回 UB）。
    然后，软件写一个循环：不断地从 UB 搬运不同的 A 矩阵块到 L0A，搬运不同的 B 矩阵块到 L0B。
    每次搬进去，就调用 Cube 算一下，结果**累加在 L0C 里面**。直到这个循环走完（K维算完），才把 L0C 的结果写回到 UB。
    *你看，虽然 Cube 内部每一拍都在疯狂流转，但从 L0 Buffer 的视角看，输出 $C$ 被死死“钉”在了 L0C 里，这就是软件实现的 **OS 策略**！*

*   **软件如何实现“宏观的 IS（输入固定）”策略？**
    如果 A 矩阵被反复使用（比如 CNN 里的同一张特征图对应不同的卷积核），软件就可以把 A 的一个大块死死锁在 **L0A Buffer** 里。
    然后不断地把不同的 B 矩阵（权重）搬进 L0B 去和它乘，算出的 C 赶快写走。
    *这时候，从 L0 Buffer 视角看，输入 A 是不动的，这就是软件实现的 **IS 策略**！*

**结论：** 硬件在 Register 级别决定了微观的效率上限（焊死以求最高频率），而**软件的调度原语是在 SRAM（L0/UB）级别排兵布阵**。编译器通过改变切块的循环顺序（Loop Reordering），决定把谁“常驻”在哪个 Buffer 里，从而实现了宏观的 IS/OS/WS。

### 3. 多核层面的 Spatial 调度 (Multi-Core Spatialization)

昇腾芯片不仅只有一个 AI Core，它有几十个 AI Core。

在编译器的语境下，`spatialize`（或 `bind`）的另一个极其重要的含义是：**将任务在芯片的多个 Core 之间进行物理空间的切分。**

这就像 CUDA 里的 `BlockIdx.x`。如果你的矩阵很大，软件可以通过 `spatialize` 原语说：
*   “把大矩阵切成 32 份，在芯片的物理空间上，散布给 32 个 AI Core 同时算。”

这也属于“空间调度”，但它是发生在一个非常宏观的芯片层面上。

*   **硬件工程师说**：“我用物理连线把最内核的 16x16 MAC 阵列焊成了 **WS（权重固定）**，因为这样布线最短、时钟频率能跑得最高。在这个微观黑盒里，你不准插手。”

### 从 2D 阵列 到 3D Cube：时间与空间的终极转换

在矩阵乘法 $C_{M \times N} = A_{M \times K} \times B_{K \times N}$ 中，有三个维度：M、N、K。

### 这“恐怖的 1 拍”在硬件内部具体是怎么发生的？

当软件调度好数据，并且 L0A 准备好了 $16 \times 16$ 的 A 矩阵块，L0B 准备好了 $16 \times 16$ 的 B 矩阵块后，硬件的“这一拍”发生了什么？

1. **超宽总线投喂（Load）：**

2. **3D 广播（Broadcast）：**
   这 256 个 A 和 256 个 B 的元素，通过复杂的金属网格网络，瞬间广播到 Cube 内部对应的 **4096 个乘法器**的输入端。
3. **4096弹齐发（Multiply）：**
   4096 个乘法器在同一个时钟周期内，**同时**完成乘法动作。
4. **加法树瞬间收敛（Spatial Reduction / Adder Tree）：**
   这是最关键的一步！在 2D 阵列里，K 轴的加法是要靠下一个周期传给下一个单元来累加的。但在 Cube 里，沿着 Z 轴（深度方向的 16 个点），硬件放了一个**加法树（Adder Tree）**。这 16 个乘法器的结果，像树根汇聚成树干一样，通过组合逻辑电路，在同一个时钟周期内瞬间相加成 1 个最终的 Partial Sum！
5. **输出（Output）：**
   时钟周期结束时，底部的 256 个结果（$16 \times 16$ 的 C 矩阵块）已经计算完毕，准备写回 L0C。

**这就是为什么它叫 Cube（立方体）！** 它将矩阵乘法的 M（宽）、N（高）、K（深）全部物理化了。

### 1. TPU 路线：脉动阵列（时间换空间，流水线）

*   **核心思想：** 细水长流，波浪式推进。
*   **做法：** K 轴（累加轴）是放在**时间**上的。数据像走迷宫一样，一个周期只在相邻的计算单元之间传递一步。
*   **优点（为什么谷歌选这个）：**
    *   **极其容易扩展！** 因为每个 MAC 单元只需要和它上下左右的 4 个邻居连线（连线极短）。由于连线短，芯片的时钟频率（Frequency）可以跑到非常高，而且你可以轻易地把阵列做得非常巨大，比如 TPUv2/v3 直接做到了惊人的 **128 x 128**。
*   **缺点：**
    *   **有预热和排空（Fill & Drain）的代价。** 就像一根长长的水管，刚开水龙头时，水要过一会儿才能流到另一头（预热）；关水龙头后，水管里的水还要排一会（排空）。如果你的矩阵非常小，水管还没填满计算就结束了，就会造成算力浪费。

### 2. 昇腾 Cube 路线：3D + 加法树（空间换时间，一拍干完）

*   **核心思想：** 暴力并拢，瞬间降维。
*   **做法：** K 轴（累加轴）全部放在**物理空间**上。16个乘法器同时乘完之后，用一个硬件“加法树（Adder Tree）”，把 16 个结果像树根一样硬生生连在一起，瞬间加出一个结果。
*   **优点（为什么华为选这个）：**
    *   **没有预热和排空的烦恼！** 它的计算是**Block（块）级别绝对确定**的。软件扔进去一个 $16 \times 16 \times 16$ 的块，它“咔嚓”一拍就全部吃掉并吐出结果。这让编译器的切块（Tiling）和调度变得极其清爽，不用去算复杂的阵列流水线排空时间。
*   **缺点：**
    *   **做不大！** 为什么昇腾的 Cube 是 $16 \times 16 \times 16$，而不是像 TPU 那样搞个 $128 \times 128 \times 128$？因为**加法树的连线太可怕了**。如果要在一个周期内把 128 个数字加起来，底层的物理导线会像蜘蛛网一样密集交错（连线延迟急剧上升），导致芯片的时钟频率根本提不上去，甚至导致布线失败（芯片造不出来）。所以 16（或者有些厂家的 32）是加法树设计的一个物理甜点区（Sweet spot）。

下面，我用一段清晰的 TensorIR 伪代码和生成的循环结构，为你硬核演示：**软件是如何仅仅通过“改变外层循环顺序（Reorder）和搬运位置（Compute_at）”，就在 L0 Buffer 上分别模拟出“输出固定（OS）”和“权重固定（WS）”的！**

### 第一步：定义基础计算和内存层级

假设我们要算一个极大的矩阵乘法 $C_{1024 \times 1024} = A_{1024 \times 1024} \times B_{1024 \times 1024}$。
而我们的 Cube 只能算 $16 \times 16 \times 16$ 的小块。

首先，我们用 TensorIR 申明计算和对应的 L0 Buffer：

```python
import tvm
from tvm import te, tir

# 1. 定义数学计算 (此时没有硬件概念，全是虚的)
M, N, K = 1024, 1024, 1024
A = te.placeholder((M, K), name="A")
B = te.placeholder((K, N), name="B")
k = te.reduce_axis((0, K), name="k")
C = te.compute((M, N), lambda i, j: te.sum(A[i, k] * B[k, j], axis=k), name="C")

# 2. 创建 TensorIR Schedule
func = te.create_prim_func([A, B, C])
sch = tir.Schedule(func)

# 3. 告诉编译器：我要在硬件上开辟三块 L0 Buffer (在 TVM 中用 "local" scope 表示)
A_L0 = sch.cache_read(sch.get_block("C"), 0, storage_scope="local") # 对应 L0A
B_L0 = sch.cache_read(sch.get_block("C"), 1, storage_scope="local") # 对应 L0B
C_L0 = sch.cache_write(sch.get_block("C"), 0, storage_scope="local")# 对应 L0C
```

---

### 第二步：将大矩阵切成 16x16 的小块 (Tiling)

```python
# 获取外层的三个循环轴 i, j, k
block_c = sch.get_block("C")
i, j, k = sch.get_loops(block_c)

# 把 1024 切成 [64, 16]。16交给Cube算，64用来做外层搬运循环
i_out, i_in = sch.split(i, factors=[64, 16])
j_out, j_in = sch.split(j, factors=[64, 16])
k_out, k_in = sch.split(k, factors=[64, 16])
```
现在我们有 6 个循环了：外层的 `i_out`, `j_out`, `k_out`（负责控制大块的搬运），和内层的 `i_in`, `j_in`, `k_in`（用来触发硬件 Cube）。

**接下来的调度，就是见证奇迹的时刻！**

---

### 方案 A：在 L0 上实现“输出固定 (Output Stationary, OS)”

如果我们希望 $C$ 矩阵的一个 $16 \times 16$ 小块死死待在 **L0C Buffer** 里，直到它的 64 次累加全做完才写出去，我们要怎么写 TensorIR 调度？

```python
# 1. 调整外层循环顺序为：i_out, j_out, k_out
sch.reorder(i_out, j_out, k_out)

# 2. 核心控制：决定数据什么时候搬入/搬出 L0
# 让 C_L0 在 j_out 级别初始化和写回
sch.compute_at(C_L0, j_out) 
# 让 A_L0 和 B_L0 在 k_out 级别往里搬
sch.compute_at(A_L0, k_out)
sch.compute_at(B_L0, k_out)
```

**👆这段 TensorIR 调度翻译成底层的 C 语言循环，长这样（极其重要，请看注释）：**

```c
// ============= 输出固定 (OS) 的底层逻辑 =============
for (i_out = 0; i_out < 64; ++i_out) {
  for (j_out = 0; j_out < 64; ++j_out) {
    // 1. [锁死输出]：在 L0C 开辟空间，准备算一个 C 的 16x16 小块
    float C_L0_buf[16][16] = {0}; 

    for (k_out = 0; k_out < 64; ++k_out) {
      // 2. [流转输入]：不断从 UB 搬新的 A 块和 B 块进 L0
      load_to_L0A(A_L0_buf, A[i_out][k_out]); 
      load_to_L0B(B_L0_buf, B[k_out][j_out]);

      // 3. [Cube爆发]：调用 Cube 指令算 16x16x16，累加到 C_L0_buf 肚子里
      Cube_MAC_16x16x16(A_L0_buf, B_L0_buf, C_L0_buf); 
    }
    // 4. 等 K 轴 64 次循环全部做完，C 的小块彻底算熟了，再写回 UB
    store_from_L0C(C[i_out][j_out], C_L0_buf); 
  }
}
```
**看明白了吗？**
因为 `k_out` 循环在最内层，所以 `C_L0_buf` 在整个 `k_out` 的循环期间**纹丝不动**（不发生跨越存储层级的读写），而 $A$ 和 $B$ 在疯狂刷新。软件完美地在 L0 层面伪装出了 **Output Stationary**！

---

### 方案 B：在 L0 上实现“权重固定 (Weight Stationary, WS)”

假设现在是算 CNN，$B$ 矩阵是卷积核权重，被复用得极其频繁，我们希望 $B$ 矩阵一旦搬进 **L0B Buffer**，就赖着不走，和不同的输入 $A$ 做计算。

TensorIR 调度只需要改**两行**代码：

```python
# 1. 改变外层循环顺序，把遍历权重的 j_out 和 k_out 放到最外面！
# 把 i_out (遍历输入的轴) 放到最里面！
sch.reorder(j_out, k_out, i_out)

# 2. 改变搬运位置
sch.compute_at(B_L0, k_out)  # B 在 k_out 搬入，待在里面
sch.compute_at(A_L0, i_out)  # A 跟着 i_out 疯狂刷新
sch.compute_at(C_L0, i_out)  # C 算完就得拿走
```

**👆这时候，底层代码的物理意义发生翻天覆地的变化：**

```c
// ============= 权重固定 (WS) 的底层逻辑 =============
for (j_out = 0; j_out < 64; ++j_out) {
  for (k_out = 0; k_out < 64; ++k_out) {
    // 1. [锁死权重]：从 UB 搬运一个 16x16 的权重块进 L0B
    // 它赖在这里不走了！
    load_to_L0B(B_L0_buf, B[k_out][j_out]); 

    for (i_out = 0; i_out < 64; ++i_out) {
      // 2. [流转输入与输出]：不断搬入新的 A
      load_to_L0A(A_L0_buf, A[i_out][k_out]);
      float C_L0_buf[16][16] = load_from_L0C(C[i_out][j_out]); // 把历史 C 拿进来

      // 3. [Cube爆发]：使用固定的 B_L0_buf 计算
      Cube_MAC_16x16x16(A_L0_buf, B_L0_buf, C_L0_buf);

      // 4. 算完当前片刻，立刻把 C 踢回 UB
      store_from_L0C(C[i_out][j_out], C_L0_buf); 
    }
  }
}
```
在这个调度下，最内层循环是 `i_out`。在 `i_out` 狂转的这 64 次里，**`B_L0_buf` 是完全不刷新的**！这就相当于软件把一块权重死死锁在了 L0B 里面，完美模拟了 **Weight Stationary**。

### 总结

在真正的工业界编译器（例如 TVM / 华为自研的 AKG 和 TIK）中，软件和硬件的分工是极其明确的：

1. **硬件**：Cube 底层的 4096 个乘法器内部怎么连线，是物理规则，改变不了。
2. **软件（TensorIR）**：通过控制**最内层执行指令的上方，到底包裹的是哪几层循环（Loop Reorder）**，以及**显式地在循环的哪个位置触发 DMA 搬运（Compute_at）**，动态决定了这一刻到底是 L0A 不动，还是 L0B 不动，还是 L0C 不动。

这就是软件调度能控制“上层存储数据流”的最真实写照！
