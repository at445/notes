## 一、双 repo 战略

### Repo 1: `npu-perf-sim`

**定位**：A perf-only, event+analytical hybrid simulator for tensor accelerators, MLIR-native, Gemmini-calibrated.

**核心技术决策**（README 第一段固定写法）：

- **Perf-only**：不维护 tensor 数据，不验证数值正确性
- **双引擎**：
  - **Analytical**：op 内部行为（MAC array cycle、DMA 单次传输 cycle），闭式公式
  - **Event**：op 间依赖、SRAM bank 仲裁、双 buffer 切换、DMA/Compute 端口竞争
- **边界判定**：有外部仲裁/决策点 → event；闭式可算 → analytical
- **MLIR-native**：吃 `my_npu` dialect IR，输出 cycle 报告
- **Gemmini 校准**：16×16 systolic, 256KB scratchpad, 单 DMA channel

**独立卖点**：

1. 业界第一个 MLIR-native 的 NPU perf simulator
2. event+analytical 混合（不是 MAESTRO 的纯解析）
3. Gemmini 校准让数字有锚

### Repo 2: `tiny-npu-compiler`

**定位**：A MLIR-based NPU compiler stack, from PyTorch to scratchpad-managed NPU IR, with closed-loop perf feedback.

**端到端流水线**：

Code

```
PyTorch (单算子或链式)
   ↓ TorchDynamo
FX Graph
   ↓ torch-mlir
linalg.matmul (MLIR)
   ↓ tiling + my_npu lowering Pass
my_npu IR (带 SRAM 分配)
   ↓ npu-perf-sim (submodule)
cycle report + MAC 利用率 + Roofline 图
```

### 关系

- `npu-perf-sim` 作为 `tiny-npu-compiler` 的 submodule
- **离线性能反馈**，不在 Pass 内同步调用（MVP 边界）
- 双向链接 README，简历按岗位类型切换主推

## 六、阶段四：NPU 后端 + 仿真器（16 周）—— 杀手锏

**核心宣言**：每天开工前读一遍——

> NPU 编译器岗位的核心矛盾是"**硬件暴露了太多细节，软件必须显式管理**"。CPU 编译器有 Cache 替你管，GPU 编译器有 SIMT 替你藏延迟，**只有 NPU 编译器需要亲自下场做物理内存规划**。阶段四的 `my_npu` 方言 + SRAM 分配 Pass + 性能仿真器 = **这个核心矛盾的最小可工作复现**。我没有真芯片，但我把这个矛盾的本质做出来了，**而且能用真实 cycle 数字证明 Pass 有效**。

### Sprint 4（Week 13~16）：杀手级优化 Pass

**目标**：用仿真器反馈，做一个**用数字证明有效**的优化 Pass。

**强烈推荐选 double buffer 插入 Pass**。理由：

- 直接对应"软件 double buffering 怎么掩盖延迟" → 你**亲手做过**
- 仿真器里效果显眼（DMA 和 compute 重叠 → cycle 显著下降）
- 工程量适中
